---
description: Run the full ServiceNow workflow on a single ingested story. Looks up stories/<STRY####>/story.md, checks dependency status in backlog.json, then dispatches to /sn-workflow with all gates intact.
argument-hint: <STRY####>
---

You are running a single ingested story end-to-end through the existing `/sn-workflow` command.

## Input
$ARGUMENTS

## Step 1 — Parse and locate

- Extract the story number from `$ARGUMENTS`. It must match `STRY\d{7}`. If missing or malformed, stop and ask.
- Locate `stories/<STRY####>/story.md` in the current working directory.
  - If the folder doesn't exist, tell the user to run `/stories-ingest <xlsx>` first and stop.
- Read `stories/backlog.json`. Confirm `<STRY####>` is present.

## Step 2 — Dependency check

Look at `stories.<STRY####>.deps` in `backlog.json`.

For each declared dep:
- Look up the dep's `status` in `backlog.json`.
- If the dep's `status` is not `done`, flag it.

If any dep is not `done`, surface the list to the user and ask whether to proceed anyway. Stop if they decline.

If the requested story's own `status` is already `done`, ask whether to re-run before doing anything.

## Step 3 — Dispatch to /sn-workflow

- Read `.claude/commands/sn-workflow.md` (the orchestrator definition).
- Read the contents of `stories/<STRY####>/story.md`. The header line (`# STRY#### — Short description`) plus the **Acceptance criteria** section is the "story description in plain English" that `/sn-workflow` expects.
- Update `backlog.json`: set `stories.<STRY####>.status = "running"`.
- Now execute every step of `sn-workflow.md` end-to-end, using the story content as the story argument. **All gates remain in place** — the user must approve between analyze / develop / test / document phases.

## Step 4 — Persist outcome

After `/sn-workflow` completes (or stops on failure / user cancel):

- If the full workflow succeeded through documentation: set `stories.<STRY####>.status = "done"` in `backlog.json`.
- If the workflow stopped with a test FAIL the user accepted: set `status = "done"` and add a `last_run_note` field with the failure summary.
- If the user cancelled or a phase errored: set `status = "blocked"` with a one-line `last_run_note`.
- Append a line to `stories/INDEX.md` is NOT needed — the index is regenerated by `parse_stories.py`. Instead, just update `backlog.json`; the index gets refreshed on the next ingest. (If you want a fresh INDEX.md now, you may run `python .claude/scripts/parse_stories.py --xlsx <last source from backlog> --out-dir stories` but only if the source xlsx path is still valid; otherwise skip.)

## Step 5 — Summary

Print:
- Story number and title
- Working folder (under `workflows/`) that `/sn-workflow` used
- Update Set name + sys_id
- ATF result
- Technical-doc path
- New status in backlog.json

## Rules
- Do NOT skip or auto-approve any of `/sn-workflow`'s gates.
- Do NOT modify `.claude/commands/sn-workflow.md` or any `sn-*` agent.
- If `/sn-workflow` itself refuses (e.g. analyst flags prod-looking instance), stop and surface to the user — do not retry.
- Touch only this story's entry in `backlog.json`. Never bulk-modify.
